# 需求分析技能 — 实体提取与数据补全

## 何时使用

当调度技能已完成意图分类（generate），且用户输入信息完整（无缺失信息），调用需求分析引擎构建项目概况。

## 行动指南

### 第1步：实体提取

从用户文本中提取四个核心实体：

- **面积提取**：支持亩（直接使用）、公顷（乘以15）、平方米（除以666.67）三种单位，正则提取数字
- **预算提取**：支持万（乘以10000）、元两种单位
- **作物类型提取**：从9种作物关键词中匹配（小麦、水稻、玉米、棉花、蔬菜、果树、大豆、花生、油菜），取第一个匹配
- **地点提取**：从城市坐标字典中匹配，优先匹配最长名称

### 第2步：实体累积合并

合并历史实体：本轮未提取到的字段使用历史轮次积累的值作为兜底。设置默认值：地点默认北京，面积默认1000亩，预算默认100万元。

### 第3步：作物自动推断

如果用户未明确指定作物类型，使用31省份→作物映射表自动推断：
- 北方省份（黑龙江、吉林、辽宁、河北、河南、山东、山西等）→ 推断为小麦
- 南方省份（江苏、浙江、安徽、江西、湖北、湖南、广东、广西等）→ 推断为水稻
- 新疆 → 推断为棉花
- 西藏 → 推断为小麦

无法推断时回退为"小麦"，并标记作物推断为True。

### 第4步：并行获取环境数据

调用数据采集模块获取两类数据：

**气象数据**：
- 解析地点坐标（覆盖150+城市）
- 请求 Open-Meteo API 的 current 参数获取实时数据
- 返回：气温(℃)、相对湿度(%)、风速(m/s)、参考ET0(mm/d)、数据源标注

**土壤数据**：
- 请求 Open-Meteo API 的 hourly 参数获取三层土壤湿度
- 计算24小时平均土壤温度和三档深度加权平均湿度
- 根据湿度推断土壤类型：大于0.4为粘土，小于0.2为砂土，0.2~0.3为粉壤土，0.3~0.4为壤土
- 返回：土壤类型、土壤温度、表层湿度、田间持水量、萎蔫系数、有效含水量

### 第5步：数据库参数查询

从数据库查询作物参数和土壤参数，查询失败时使用硬编码默认值：

**作物参数**：Kc系数（4个生育期）、根系深度、各生育期天数
**土壤参数**：田间持水量、萎蔫系数、有效含水量

### 第6步：构建项目概况

返回结构化项目概况字典，包含：地点、作物类型、面积、预算、气象数据、土壤数据、作物参数、土壤参数、原始输入、作物是否推断标记。

## 决策标准

| 场景 | 决策 |
|-----|------|
| 作物已提取 | 使用用户指定的作物 |
| 作物未提取 + 地点可匹配省份映射表 | 自动推断 + 标记推断 |
| 作物未提取 + 地点无法匹配 | 回退为"小麦" |
| Open-Meteo API调用失败 | 返回默认值（温度25℃、湿度60%、ET0=3.5mm/d） |
| 数据库查询失败 | 使用硬编码默认作物/土壤参数 |
| 坐标未匹配到城市字典 | 降级使用北京坐标 |

## 资源文件

| 资源 | 说明 |
|-----|------|
| DataAcquisition.get_weather_data() | async调用，输入地点，输出气象数据字典 |
| DataAcquisition.get_soil_data() | async调用，输入地点，输出土壤数据字典 |
| 城市坐标字典 | 本地查找，输入地点字符串，输出纬度和经度 |
| MySQL CropParameter | SQLAlchemy查询，输入作物名称，输出Kc系数/根深/天数 |
| MySQL SoilParameter | SQLAlchemy查询，输入土壤类型，输出田间持水量/萎蔫系数 |

## 质量检查

在输出项目概况前确认：
- 地点和作物类型不为空（允许推断值）
- 面积为float类型，单位统一为亩
- 预算为int类型，单位统一为元
- 气象数据包含温度/湿度/风速/ET0四个字段
- 土壤数据包含土壤类型/田间持水量/萎蔫系数/有效含水量
- 作物参数包含4个生育期的Kc值和阶段天数
- 作物推断标记正确（用户指定为False，推断为True）
