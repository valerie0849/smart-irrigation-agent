# 调度技能 — 意图路由与流程编排

## 何时使用

此技能是系统的入口调度器，位于后端路由处理中。每个用户请求到达时首先执行意图分类。

## 行动指南

### 第1步：意图分类

按以下优先级判定用户意图：

**优先级1：modify（最优先）**
- 触发条件：用户输入包含修改方案、调整方案、优化方案、帮我把、改一下等关键词

**优先级2：weather**
- 触发条件：用户输入包含天气、气象、气温、降水、降雨、湿度等关键词

**优先级3：soil**
- 触发条件：用户输入包含土壤等关键词

**优先级4：knowledge_browse**
- 触发条件：用户输入包含查看知识森林、知识列表、浏览知识等关键词

**优先级5：knowledge**
- 触发条件：用户输入包含知识、依据、标准、规范、参考标准等关键词

**优先级6：generate（显式触发词）**
- 触发条件：用户输入包含生成方案、设计方案、制定方案、帮我设计等关键词

**优先级7：generate（隐式检测）**
- 检测到地点+作物 → 判定为generate
- 检测到地点+面积/预算 → 判定为generate
- 检测到面积+作物 → 判定为generate
- 仅检测到面积或预算 → 判定为generate

均未匹配 → 判定为unknown

### 第2步：实体提取与累积

从用户输入中提取四个核心实体：地点、作物类型、面积、预算。将提取结果合并到会话级累积字典中，每轮对话提取的新实体与历史实体合并，本轮未提取到的字段使用历史值。

### 第3步：参数微调检测

当意图为generate时，检测是否应转为modify：
- 条件A：对话历史中存在长度大于500字符的助手回复（表示已生成过方案）
- 条件B：本轮未提取到地点/作物类型/面积（用户未改变项目基本信息）
- 条件C：提取到预算或包含修改类关键词（预算、万、修改、调整、改成、改为、换成等）

同时满足条件A、B、C → 将意图从generate转为modify。

### 第4步：信息完整性检查

当意图为generate时，检查是否缺失必要信息：
- 缺失地点 → 提示用户提供项目地点
- 缺失作物类型 → 提示用户提供作物类型
- 缺失面积 → 提示用户提供灌溉面积
- 缺失预算 → 提示用户提供预算范围

存在缺失信息时，向用户追问补全信息，不启动方案生成流程。

### 第5步：流程路由

根据最终意图路由到不同处理逻辑：
- knowledge_browse → 返回知识森林全貌概览
- knowledge → 返回知识检索结果
- weather → 调用数据采集获取气象数据
- soil → 调用数据采集获取土壤数据
- generate（信息完整）→ 启动SSE流式方案生成
- generate（信息缺失）→ 返回追问信息
- modify → 启动SSE流式方案修改
- unknown → 有灌溉相关词则追问，无则兜底回复

## 决策标准

| 场景 | 决策 |
|-----|------|
| 首次对话，无历史方案 | 正常意图分类，检测缺失信息 |
| 有历史方案，用户只改预算/参数 | 切换为modify意图 |
| 有历史方案，用户提供新地点/作物 | 保持generate意图（全新方案） |
| 知识森林未构建 | generate流程中自动触发构建 |
| LLM不可用 | 降级使用模板方案 |

## 质量检查

在每次路由后确认：
- 意图分类正确（7种意图分属合理）
- 参数微调检测逻辑已执行（当intent=generate且有历史方案时）
- 实体已累积到会话字典
- 对话历史已保存到数据库
- 缺失信息已列出（地点/作物/面积/预算）
