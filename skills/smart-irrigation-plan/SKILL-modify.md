# 方案修改技能 — 定向修改与一致性保障

## 何时使用

当调度技能检测到参数微调（有历史方案+无新地点/作物+仅改预算等参数）或用户直接发出修改意图时，调用方案修改引擎。

## 领域知识

### 方案章节结构

方案包含5个章节模块，按顺序排列：

| 模块key | 章节标题 | 内容 |
|--------|---------|------|
| overview | 一、项目概况 | 项目基本信息、气象土壤数据 |
| irrigation | 二、灌溉制度设计 | ET0、Kc系数、灌水计划 |
| equipment | 三、设备选型清单 | 设备列表、成本核算 |
| references | 四、溯源依据 | 知识来源、标准引用 |
| explanation | 五、方案说明 | 方案总结与解释 |

### 章节依赖关系

各章节之间存在上下游依赖关系：

- **overview**：无依赖（根节点）
- **irrigation**：依赖 overview
- **equipment**：依赖 irrigation + overview
- **references**：依赖 irrigation + equipment
- **explanation**：依赖 overview + irrigation + equipment

反向依赖：修改某章节时，其依赖方和依赖它的所有章节都需要同步更新。

## 行动指南

### 第1步：LLM解析修改意图

将用户修改请求送入大语言模型，解析为结构化JSON，包含：
- **operation**：操作类型（update_budget/replace_method/add_section/delete_section/update_parameter）
- **target_sections**：直接受影响的章节key列表
- **constraints**：约束参数（如新预算值）
- **dependency_hint**：可能被连带影响的章节key

**回退解析**：LLM失败时使用关键词匹配。预算/投资→overview+equipment；面积/作物→overview+irrigation；滴灌/喷灌→equipment+irrigation；传感器/水泵→equipment；灌水定额/Kc/ET0→irrigation；标准/规范→references。

### 第2步：BFS依赖展开

根据章节依赖关系图，通过广度优先搜索自动识别所有受影响的章节：
- 从目标章节出发，向上游追溯依赖
- 向下游追溯反向依赖
- 按章节顺序排序输出

示例：修改equipment → 影响 overview, irrigation, equipment, references, explanation

### 第3步：并行知识检索

为每个受影响章节定制检索查询，并行从知识森林获取相关知识包：
- overview：灌溉项目概况、投资构成、预算编制
- irrigation：灌溉制度设计、灌水定额、Kc系数、Penman-Monteith
- equipment：灌溉设备选型、配置方案、成本核算
- references：灌溉标准规范、GB/T、SL/T、FAO溯源依据
- explanation：灌溉方案说明、注意事项

### 第4步：骨架-填充式分章生成

按章节顺序逐个生成新内容：

1. 获取上文章节（保持衔接）
2. 获取下文章节（保持衔接）
3. 格式化参考知识
4. 提取关键约束参数（突出显示，优先级最高）
5. 使用LLM生成新章节：先输出1-2级小标题骨架，再逐节填充内容
6. 保持与原方案相同的编号层级和术语风格
7. 关键参数标注来源
8. 生成失败（结果为空或过短）→ 保留原章节

### 第5步：合并方案

采用三级替换策略将新章节合并到方案中：

1. **精确匹配替换**：原内容在方案中 → 直接字符串替换
2. **正则定位替换**：通过章节标题正则定位 → 替换该章节区域
3. **兜底追加**：无法定位 → 追加到全文末尾

合并后验证交叉引用有效性。

### 第6步：局部质量审核

仅审核被修改的章节，检查：

**合规性**（初始100分）：灌溉水利用系数、灌水定额、标准引用
**合理性**（加分制）：Penman-Monteith公式、Kc系数、灌水定额、灌水频次

同时执行整体一致性检查：
- 预算一致性：验证全文预算数据统一
- 灌溉方式一致性：设备清单与灌溉方式匹配
- 方案长度：新方案长度 ≥ 原方案 × 0.7

### 第7步：质量迭代

评分 < 75 分时，将问题反馈给LLM重新生成（最多重试1次）。

### 第8步：Diff对比生成

对比新旧章节内容，提取：
- 新增词汇（最多5个）
- 删除词汇（最多5个）
- 行数变化

### 兜底知识依据

当知识森林无返回结果时，生成兜底知识引用，引用灌溉设计规范GB/T 50363-2018和SL 287-2014标准。

## 决策标准

| 场景 | 决策 |
|-----|------|
| LLM解析意图失败 | 使用关键词回退解析 |
| 知识森林不可用 | 使用兜底模板知识依据 |
| 章节标题正则不匹配 | 兜底追加到方案末尾 |
| LLM生成结果过短（≤20字符） | 保留原章节内容 |
| 质量评分 ≥ 75 | 正常返回 |
| 质量评分 < 75 | 自动重试一次，将问题作为修正指令 |
| 重试后仍 < 75 | 返回当前结果 + 重试信息 |

## 资源文件

| 资源 | 说明 |
|-----|------|
| DeepSeek LLM | API调用，输入提示词，输出生成文本 |
| KnowledgeForest.query() | async调用，输入查询字符串和k=3，输出知识检索结果列表 |
| 正则匹配 | 本地实现，输入方案文本，输出章节定位、参数提取 |
| BFS算法 | 本地计算，输入目标章节列表，输出受影响章节集合 |

## 质量检查

在输出前确认：
- 受影响章节全部更新
- 新方案长度 ≥ 原方案长度 × 0.7
- 预算数据在全文一致
- 灌溉方式与设备清单一致
- 质量评分 ≥ 60（低于60需人工介入）
- diff数据包含所有实际修改的章节
- 知识依据不为空

## 输出规范

修改结果包含：新方案内容、原始方案、质量评分和等级、一致性警告、受影响章节列表、知识依据、diff对比数据、重试信息、各章节新内容。
